# 在 Luffa SuperBox 上做一本「互动书」

> 把你最爱的非虚构书,变成读者能动手玩的小程序。
> 8-12 小时,AI Coder 帮你做 80% 的工作。

---

## 为什么要做这件事

想象一下:你读完一本让你激动的非虚构书 —《金融周期》《思考快与慢》《精益创业》《原则》。合上书的那一刻你觉得"懂了",但三周后再被人问起,你只剩一两句口号还在脑子里。

这是因为**阅读是被动的**。书里的每个观点、每张图表、每个反直觉的事实,在你眼前流过去时确实点亮了你的某个神经元,但没有"用过"的知识不会留下。

互动书想解决的就是这件事。

一本书不再是静态的文字,而是一个**让你能动手把书里的论点玩出来的小程序**:

- 改一个参数,看看作者的模型在什么条件下会**失效**。
- 做一道选择题,发现自己的直觉**错在哪**,以及为什么大多数人都错。
- 用今天真实的市场数据,跑一下作者 2020 年写下的那个判断,**它现在还成立吗?**

读完之后,书里的智慧才真正从纸上**长进**你的脑子里。

这份指南讲的是怎么从零开始做出这样一本互动书,部署到 Luffa SuperBox 上,让任何人都能扫码玩。整个过程 8-12 小时,前提是你愿意把一半时间花在认真想"做什么内容"上,另一半交给 AI Coder 写代码。

## 这份指南为谁写

### 👨‍💻 AI Coder 用户(用 Claude Code / Cursor / Windsurf 等)

你已经习惯让 AI 帮你写代码。但"做一个完整产品并上线"和"让 AI 写一个 demo"是两回事。这份指南会告诉你:

- **AI 能帮你做到哪一步**:几乎所有的页面代码、数学函数、API 集成。

读完后你会有一份**完整心智模型**,下一次拿到一本想做的书,你能立刻知道"问 AI 什么、自己做什么、什么时候喊停"。

### 📚 爱读书的技术小白

你不需要写过一行代码,但你想知道:有没有可能,把那本一直让你念念不忘的书,变成一个能给朋友/学生/同事玩的工具?

这份指南**不会逼你学 React 或 JavaScript**。它会用大白话告诉你:

- **一本互动书的"骨架"长什么样**:模拟器是什么、反预期事件怎么设计、为什么需要章节地图。
- **哪些事情可以放心交给 AI 做**:写代码、做页面、对接 API。
- **哪些事情你必须自己想清楚**(没人能代劳):书里最有价值的 3 个工具 / 公式 / 框架是什么?每章哪个事实最反直觉?读者按完模拟器之后应该"哇"在哪里?
- **一些不让你被 AI 绕晕的常识**:为什么 API key 必须放在"代理"里、为什么打开率受"包体大小"影响、什么样的描述能通过 Luffa 审核。

读完后你不会变成程序员,但你会变成一个**能指挥 AI Coder 做出一本互动书的产品经理**。这是这个时代最有杠杆的角色之一。

## 总览 · 互动书的心智模型

一本互动书等于三样东西的组合:**内容** + **模拟器** + **反预期事件**。

**内容**是书的章节地图和摘要 —— 但不是把整本书塞进小程序。互动书的核心价值是"重写",不是"复刻"。如果用户想完整读这本书,他应该去买实体书。

**模拟器**是把作者的核心思考工具变成可玩的计算器。一本财经书有 NPV/IRR/WACC;一本心理学书有"框架效应模拟器";一本商业书有"单元经济学计算器"。模拟器的关键不是漂亮,是**让用户改参数后看到反直觉的结果**。

**反预期事件**是每一章里那个"以为是 A,其实是 B"的核心事实。配上一道三选项判断题 + 答错的解释,读者就在 30 秒内同时被打脸 + 长记性。这是互动书相比纯文本的最大教育优势。

### 三个不能违反的原则

1. **不要把整本书塞进去**。互动书的价值在"重写为可玩",不在"在小屏幕上再读一遍"。
2. **模拟器必须反预期**。如果用户改参数后结果跟他猜的差不多,他根本不会改。设计模拟器时,**把"令人意外的输出"当目标**,而不是"功能完整"。
3. **API key 永远不在小程序包里**。所有外部 API(FRED 经济数据、OpenAI、股价等)都要走云端代理。小程序包会被反编译,密钥放进去等于公开。

# Phase 0 · 内容设计

> 这是整个流程里**最不能交给 AI** 的一步,也是**花时间最值**的一步。

省这一步的代价:做出来的小程序"能跑但没用",用户打开 30 秒就关掉。

读完书(电子版/纸质都行)后,在白纸上写三件事:

### 0.1 三个"核心数学 / 概念框架"

这本书的工具箱里,最常用的 3 个**操作**是什么?注意是操作,不是概念 —— 概念是"机会成本",操作是"NPV 计算"。

举例:

- **《金融周期》(Oppenheimer)** → NPV / IRR / WACC(估值与资本预算)
- **《精益创业》(Ries)** → MVP 假设清单 / 单元经济学 / 转化漏斗
- **《思考快与慢》(Kahneman)** → 框架效应模拟 / 锚定测试 / 概率直觉测试
- **《原则》(Dalio)** → 决策日志 / 可信度加权 / 反思矩阵

每个操作对应一个模拟器页面。**最多 3 个**,贪多就稀释了。

### 0.2 每章一个反预期事件

翻完每一章后问自己:**这一章里有什么是大多数读者会"以为是 A,其实是 B"的?**

要求:

- 必须有书里**具体的数据 / 表格 / 历史案例**支撑。这是和"心灵鸡汤"的分界线。
- 写成一句"钩子句"(hook),够短够冲击 —— 例:"经济最差的时候是最差的买点 —— 错。"
- 配一段详细解释,把数据 / 案例完整摆出来。
- 设计一道三选项判断题,让用户在被告知"真相"之前先猜一次。
- 写好"答错的解释",讲清为什么大多数人都猜错。

11 章书就有 11 个反预期。这是互动书里**单点价值密度最高**的内容。

### 0.3 章节地图

把这本书的所有章节做成结构化清单。每一章包含:

- 章节号
- 所属"大部分"(如果书分了 part)
- 章名
- 一两句话的摘要
- 这一章的反预期事件(hook + body + quiz + 答错解释)

这份清单是后面 AI 写代码时最重要的"原料表"。

### 0.4 给模拟器设计「情景模板」

光给一个空白计算器,用户不知道怎么玩。**预设 4-6 个故事化情景**,每个对应书里的一个论点。点一下情景按钮,所有参数自动填好,用户立刻看到该情景下的反差。

以《金融周期》的 NPV/IRR 模拟器为例:

| 情景名 | 故事 | 教学点 |
|---|---|---|
| 希望期反弹 | 衰退底部入场,1 年后强劲反弹 | IRR ≈ +24%,对应原书表 4.1 希望期 +69% |
| 增长期成熟 | 盈利到顶,逐年放缓 | IRR ≈ +0.4%,对应表 4.1 增长期 +2% |
| 泡沫追高 | 500 倍 PE 买龙头股 | IRR ≈ −33%,漂亮 50 的命运 |
| 科技革命赢家 | 前期亏损,后期指数级回报 | IRR ≈ +14%,长期赢家但要忍前期 |

设计原则:**情景间的结果跨度要大**(本例 −33% 到 +24%),用户切换时能立刻看到反差。数字尽量对齐书里的真实数据(本例 +0.4% ≈ 表 4.1 增长期 +2%),让模拟器和书形成强对照。

### 0.5 这一步要花多少时间?

读完一本约 250 页的书加上 Phase 0 全过程,大约 **4-6 小时**。

**这是整个项目里最值得的时间**。把这一步草草做完,后面所有代码都救不回来。

# Phase 1 · 注册 Luffa 控制台,拿 AppID

> 一次性,约 10 分钟。AI 帮不了你,纯手工。

打开 Luffa Cloud 开发者控制台,右上角"注册账号"。注册完登录后,点"创建小程序",填几件事:

- **名称**:中文 ≤ 20 字符。比如"金融周期 · 互动学习"。
- **分类**:选"教育"或"工具"或"阅读"。
- **描述**:≥ 50 字,要说清"是什么 / 为谁 / 做什么"。这段会进审核,认真写。
- **图标**:512×512 PNG。可以让 AI 帮你画一个简洁的几何图标。

控制台会自动发一个 AppID,形如 `mpkboyj2fc3l4u7q`(16 位字母数字)。**复制保存** —— 这是接下来 IDE 里唯一的"识别码"。

**关键提醒**:这一步**不要**马上配域名白名单。如果你打算用实时数据(Phase 5),要等代理部署完拿到真实 URL 后再配。现在配会指向不存在的地址。

# Phase 2 · 项目脚手架

> 30 分钟。AI 主导,你做技术决策 + 审查。

### 2.1 第一个决策:native 还是 Taro?

| | Native (WXML/WXSS/JS) | Taro 4 + React + TypeScript |
|---|---|---|
| 包体 | 极小(本项目约 200 KB) | 较大(600 KB 起) |
| 学习曲线 | 几乎为零 | 已经会 React 才舒服 |
| 适合 | 内容型 / 数学型应用 | 复杂 UI / 现有 React 库要复用 |

**互动书优先选 Native**。Taro 的 runtime 开销对一个学习型应用是浪费。让 AI 知道你选了 native,它就不会引入 React/Vue/TypeScript 等不必要的层。

### 2.2 一个小程序长什么样

最终的项目目录里,你会看到:

- 一个**根配置文件**,告诉小程序"我有哪些页面、底部 tab bar 是什么样的、整体配色如何"。
- 一个**全局逻辑文件**(app.js),负责跨页面的状态,比如用户答题进度的本地缓存。
- 一个**全局样式文件**(app.wxss),定义品牌色、字体、卡片样式。
- 一个 `utils/` 文件夹,放可被多个页面复用的工具:数学函数(NPV / IRR / WACC)、缓存(本地存储 + TTL)、Luffa 原生桥接、外部 API 客户端。
- 一个 `data/` 文件夹,放 Phase 0 设计的章节地图和情景模板。
- 一个 `pages/` 文件夹,放**六个页面**:首页(目录 + 实时仪表盘)、3 个模拟器、1 个反预期总览、1 个章节详情子页。

### 2.3 tabBar 的硬限制

底部 tab bar **最多 5 个**(SuperBox 和微信都一样)。本项目的分配:目录 / 模拟器 1 / 模拟器 2 / 模拟器 3 / 反预期总览。章节详情是从首页或反预期页跳过来的子页,不占 tab。

设计建议:
- **不放 iconPath**,只用文字。省一堆 PNG 资源,符合"包体越小越好"原则。
- 配色和品牌色对齐:tabBar 文字色暗、选中色亮(本项目暗灰 + 金色)。

### 2.4 三个配置文件的角色

- `project.config.json` —— **IDE 配置**。最关键的是 AppID 字段(Phase 1 复制保存的那串)。另外有个"是否检查域名白名单"开关,在 IDE 里调试时可以临时关,但**真机一定要为 true**。
- `app.json` —— **小程序自己的配置**。告诉运行时:"我有这 6 个页面要注册"、"tabBar 显示哪 5 个"、"全局窗口背景色和字体颜色"。
- `sitemap.json` —— **搜索引擎索引规则**。互动书一般全部允许,一行配置搞定。

让 AI 一次性生成这三个文件 + 上面 2.2 提到的目录结构,然后你打开 Luffa Cloud-Devtools 试着"打开项目"指向这个目录。如果模拟器能渲染出空首页,Phase 2 就过了。

# Phase 3 · 数学和数据

> 1-2 小时。AI 写代码,**你必须亲自验证数学**。

### 3.1 数学要"纯净"

让 AI 把所有数学(NPV / IRR / WACC 之类)写在一个单独的工具文件里,**只接受数字和数组、只返回数字和对象,绝不调用小程序专有 API**。

这样做有两个好处:第一,你可以在自己电脑上用 Node 直接跑这个文件,不用启动整个小程序就能测;第二,数学和 UI 解耦,以后改 UI 不会动到公式,改公式也不会动到 UI。

### 3.2 让数学"感知周期阶段"(关键技巧)

光实现"教材版"的 NPV 远远不够。互动书的高级感来自**同一公式在不同语境下输出完全不同的结果**。

具体做法:让 AI 给每个"周期阶段"(绝望 / 希望 / 增长 / 乐观)各定义一个折现率的"加成"。比如绝望期投资者要求更高回报,加 4 个百分点;希望期市场愿意接受更低折现率,减 2 个百分点。

这样,NPV 页就能在一张表里展示**同一现金流在 4 个阶段下完全不同的 NPV**。用户的直觉里"现金流不变 NPV 就不变",但现在他亲眼看到 NPV 在 4 个阶段间从正翻成负 —— **抽象的"周期红利"瞬间变成了 4 个数字**。这就是互动书的"哇"时刻。

### 3.3 在 Node 里测一遍数学(**这步必做**)

让 AI 写完数学函数后,**你自己在命令行里跑一道教材标准题**。比如:

- "初始投入 100,后续 5 年每年回 30,折现率 10%" —— 教材标准答案 NPV = 13.7236。
- 同一组现金流的 IRR 教材标准答案 = 15.24%。
- WACC 标准例题:股权占比 60%、债务 40%、股权成本 12%、债务成本 6%、税率 25% —— 标准答案 9.00%。

把 AI 写的代码跑一遍,确认每个数字和教材都精确对得上(差到小数点后第 4 位)。

**这是唯一能在小程序之外捕获数学 bug 的机会**。本项目就靠这一步发现了"增长期成熟"情景的 IRR 算出来是 −13.5%,与设计意图"经济仍在增长,IRR 应略正"严重矛盾。改了现金流参数才纠正过来。

如果你不亲自验,AI 生成的代码即使语法没错,数学也可能微妙地错 —— 而错的数学在用户那里会**永远错下去**,你的产品就废了。

### 3.4 章节数据兼任「参数库」

让 AI 把 Phase 0 设计的章节数据放在 `data/chapters.js` 里,这个文件同时给两类页面服务:

- **首页 / 反预期总览 / 章节详情** 用它来渲染章节内容。
- **三个模拟器** 用它的另一部分(`SCENARIOS` 对象)拿到预设情景。

这样章节和模拟器自然形成**交叉引用**。用户在第三章详情页底部点 "→ NPV" 按钮,可以直接跳到 NPV 模拟器并自动加载"希望期反弹"情景。这种联动是"互动书"和"普通计算器 app"的最大差别。

# Phase 4 · 页面

> 3-5 小时。AI 主力,你定 UI 优先级和体验细节。

### 4.1 模拟器页的通用六段式

每个模拟器页,从上到下都是这六段:

1. **概念卡片** —— 一段话讲清这个模拟器对应的公式 / 框架,引用书里的章节。
2. **情景模板按钮** —— 4-6 个 Phase 0.4 设计的故事化情景,一键切换会重置所有参数。
3. **参数输入** —— 滑块、数字输入框、表格行编辑等。让用户随便改。
4. **实时计算结果** —— 一个**大号金色数字**,任何参数变化都立刻重算。
5. **对照表格** —— 一张表把不同周期阶段 / 不同情景的结果并排放,让"反预期"显形。
6. **关键观察** —— 一段**根据当前结果动态生成**的文字解读,像"作者站在用户旁边讲"。

**实现要点**:任何参数变化就立刻重算 —— 不要做 throttle / debounce。这种 app 的计算几乎都在 1ms 以下,直接全量重算最干净,延迟反而让人觉得卡。

### 4.2 「关键观察」是这本互动书最具杀手锏的一段(优先做好)

让 AI 写一个简单的判断逻辑:用户当前的参数组合属于哪一类?

- "全周期都创造价值" —— 显示某条来自书里的洞察(比如:"巴菲特意义上的好生意")。
- "全周期都不创造价值" —— 显示另一条(比如:"不要靠等周期反转自我安慰")。
- "不同周期间跨度极大" —— 显示第三条(比如:"这是高久期资产,估值几乎只是折现率的函数")。
- "正负翻转,取决于入场点" —— 显示第四条(比如:"这是周期红利依赖型项目,入场点决定一切")。

这一段是把**作者的智慧灌进交互**的关键 —— 远比写一段静态说明文字有效。用户每改一次参数,都像收到一条作者的私信点评。

### 4.3 章节详情页

章节详情页是从首页或反预期总览页**跳过来**的子页,结构很固定:

1. 章节标题 + 摘要(从 chapters.js 拉)。
2. 反预期 hook(用紫色 / 高亮卡片突出)。
3. 详细解释 + 数据 / 案例完整呈现。
4. 三选项判断题。**选错变红,选对变绿**,然后才显示"为什么"的答错解释。
5. 上一章 / 下一章导航,方便连续读。
6. **联动到三个模拟器的按钮** —— 让用户读完反预期立刻去模拟器验证。

第 4 步的"先猜再揭晓"是这个页的精华,**不能让用户跳过猜的环节直接看答案**,否则反预期效果减半。

### 4.4 进度持久化

让 AI 用小程序自带的本地存储 API,记下用户答过的每道题、选了哪个选项。下次打开时不重置 —— 用户能看到自己已经答过哪几章、选错了几次。

这是任何"学习类应用"的最低体验。也是后续可以扩展"答对超 N 题解锁徽章"功能的数据基础。

### 4.5 Luffa 原生桥接

即使你的 MVP 用不到钱包 / 群分享 / 链调用,**也先让 AI 包一层 Luffa 桥接函数**。

未来你想加"完成答题领 NFT 徽章"、"一键分享某章反预期到 Luffa 群聊"、"用 Luffa 身份登录"等功能时,改一行调用就行,不用重构整个项目。这是几行代码的代价,换来未来零摩擦升级。

# Phase 5 · 实时数据集成(可选)

> 30-60 分钟。互动书的"活"靠数据 —— 让模拟器跟着今天的市场跑,体验完全不同。

可以跳过这一步,你的互动书也能正常工作。但如果你的书涉及"市场 / 经济 / 实时数据",接上实时数据后的版本和静态版本不是一个量级的。

### 5.1 为什么必须走「代理」?

互动书的核心红线:**API key 永远不能放在小程序包里**。小程序包是公开的、会被反编译。一旦 key 进了包,任何人都能拿去用 —— 几天内你的免费 quota 就被刷爆,或者被收高额账单。

正确做法:

```
你的小程序  →  你部署的代理(在云端持 API key) →  外部 API(FRED 等)
```

代理是一个极简的 Web 函数,跑在你自己的免费云服务器上(Vercel / Cloudflare Workers),它持有 key、检查请求合法性、转发到 FRED、返回结果。**小程序看不到 key,只看到代理 URL**。

### 5.2 用 Vercel 部署代理(推荐路径)

#### 准备

让 AI 在仓库根目录新建一个 `api/` 文件夹,里面放一个文件 `fred.js`(对应 FRED API,如果你用别的 API 就换名字)。这个文件就是一个 Web 函数,做四件事:

1. 接收前端传来的 series 参数(比如 "DGS10" = 10 年期美债收益率)。
2. **检查 series 是不是在白名单内** —— 即使有人拿到代理 URL 也只能拉预设的几个数据。
3. 用环境变量里的 API key 去 FRED 抓数据。
4. 加一个 1 小时边缘缓存,然后返回 JSON。

三个防御要全有:**白名单** + **CORS 全开**(小程序请求不带 origin,全开无风险)+ **边缘缓存**(节省 API quota)。

#### 部署步骤(纯手工,AI 帮不了)

1. **申请 API key**。比如 FRED 在 https://fred.stlouisfed.org/docs/api/api_key.html 上注册免费拿,32 位字符串。
2. **把代码推到 GitHub**。让 AI 帮你跑 `git init / commit / push`。如果是第一次用 GitHub,先在网页上建一个空仓库。
3. **打开 vercel.com,用 GitHub 账号登录**。
4. **点 "Add New → Project"**,选刚 push 上去的仓库,Import。
5. **配置页面**:Framework 选 "Other",其他都留空。**不要立刻点 Deploy**。
6. **展开 "Environment Variables"**,加一项:
    - Name 填:`FRED_API_KEY`(全大写、下划线、没有空格)
    - Value 填:你第 1 步拿到的 32 位 key
    - Environment 三个全勾(Production / Preview / Development)
7. **点 Deploy**。约 30 秒后 Vercel 给你一个 URL,形如 `https://你的项目名.vercel.app`。

#### 验证代理通了

打开浏览器访问 `https://你的项目名.vercel.app/api/fred?series=DGS10`,如果看到一段 JSON 含 `observations` 数组(里面是日期 + 数值的对象),代理就通了。

#### 两个会让人困惑半天的坑

**坑 1:返回 "FRED_API_KEY env not set"**

99% 的情况是 env 加了但函数没重新部署。Vercel **不会自动 redeploy 现有函数**,你要么手动去 Deployments 找最新那条点 Redeploy,要么让 AI 帮你 push 一个空 commit 触发自动 redeploy(这是本项目的做法 —— 干净不出错)。

**坑 2:浏览器报 SSL 错误**

短暂的 `SSL_ERROR_SYSCALL` 通常出现在 Vercel 正在切换部署版本的几秒钟。等 5-10 秒重试。

### 5.3 小程序端连接代理 —— 三件事不能少

让 AI 在 `utils/` 里写一个外部 API 客户端,务必做到三件事:

1. **本地缓存 12 小时**。FRED 是日级 / 月级数据,不需要每次都重新拉。配合 Vercel 的边缘缓存 1 小时,真正打到 FRED 的请求量几乎为零,quota 用不完。
2. **优雅降级**。如果 PROXY_BASE 还是占位 URL(说明你还没填上 Vercel 给的实际地址),UI 显示"未配置代理"提示,模拟器照常工作。**绝对不能让 UI 因为外部 API 挂掉而崩**。
3. **错误隔离**。多个 series 并行请求时,某一个挂了不能影响其他 5 个。

### 5.4 用实时数据让书的论点"在用户眼前发生"

最值得做的一件事:在首页 hero 卡片下方,加一个**实时仪表盘**,渲染 6 个核心宏观指标(《金融周期》案例里是 10 年期美债收益率、2 年期美债收益率、收益率曲线利差、联邦基金利率、失业率、CPI 同比)。

进阶招:在客户端加一段判断 —— 如果"10 年期 - 2 年期"利差为负(收益率曲线倒挂),自动弹出书里第六章关于熊市指标的解读。

**这就是用今天的市场,让作者的 2020 年论点"活"在用户眼前**。比任何静态文字都有冲击力。

# Phase 6 · 部署到 Luffa

> 30 分钟。你必须人工做,AI 帮不上忙。

### 6.1 配域名白名单(必做)

打开 Luffa 控制台 → 你的小程序 → 开发设置(或称"服务器配置")→ "request 合法域名" 列表。

把你 Phase 5 拿到的 Vercel URL 加进去,**注意几点**:

- 只到域名,**不带路径**。填 `https://你的项目名.vercel.app`,不要填 `/api/fred` 后缀。
- **必须是 HTTPS**,HTTP 一定被拒。
- 加进去之后**等约 1 分钟**让 Luffa 控制台缓存刷新,不然 IDE 里立刻测会失败。

如果跳过这一步,你的真机用户**100% 会看到白屏 / 加载失败**。IDE 模拟器即使勾"不校验合法域名"也只是绕过 IDE,真机一定走白名单。

### 6.2 在 Luffa Cloud-Devtools 里上传

1. 从 Luffa 控制台下载 Cloud-Devtools(Mac/Windows/Linux 三个平台都有)。
2. 用注册账号登录(必须是同一个账号)。
3. **打开项目** —— 选 `miniprogram/` 目录(Native 路径)。AppID 用下拉选,不要手动输,避免手敲错误。
4. 在 IDE 里做这几件事:
   - **预览** —— 扫码用真机过一遍核心路径:
     - 首页所有章节卡片能点开吗?
     - 模拟器切换情景后表格实时更新吗?
     - 滑块拖动结果跟着变吗?
     - 选错答案变红、选对变绿吗?
     - tabBar 切换无白屏吗?
     - 实时仪表盘有真实数据吗(域名白名单生效后)?
   - **包体检查** —— IDE 显示的包大小必须 **< 2 MB**(SuperBox 硬限制)。本项目约 200 KB。
   - **上传** —— 填版本号(建议 `1.0.0`)+ 更新说明 → 提交。

### 6.3 常见上传错误

| 错误 | 原因 | 解决 |
|---|---|---|
| AppID 不匹配 | project.config.json 里 TCMPPappid 不对 | 从控制台**复制粘贴**,不要手敲 |
| 包体超过 2 MB | 引了 lodash / moment / 图片 | 让 AI 改成按需引用,删图片资源 |
| "wx.invokeNativePlugin is not a function" | 不在 Luffa IDE 内运行 | 这是预期 —— 你的 Luffa 桥接代码应该有 try/catch 降级 |

# Phase 7 · 审核 + 发布

> 1-3 个工作日。等 Luffa 审核员。你的工作是**把审核需要的话术写对**。

### 7.1 审核前在控制台填的内容

- **功能分类**:教育 / 工具 / 阅读。
- **测试账号**:互动书一般不需要(无登录),勾"无需测试账号"。
- **隐私合规**(这一项最容易被打回!):
  - 写明"本应用不收集任何用户个人数据"。
  - 写明"答题进度仅在本地存储,不上传服务器"。
  - 如果你接了实时 API:"使用 [FRED 等] 公开数据 API,通过我方代理调用,不传输任何用户数据"。
- **金融建议免责声明**(教育 / 财经类**必加**):
  - "本应用基于公开出版的《[书名]》一书的互动学习,不构成投资建议"。
  - "示例情景与历史数据均来源于公开资料,不构成对任何标的的预测"。

写得**越具体越保守**,审核越顺利。

### 7.2 提交审核

控制台 → 版本管理 → 找到你刚上传的"开发版本" → 点"提交审核"。

等 1-3 个工作日。如果被打回,看具体原因,改完重传重新提交即可。常见打回原因都在附录 A 的速查表里。

### 7.3 发布

审核通过后,状态变成 "Approved"。团队管理员点"发布",状态变成 "Online"。

控制台会给你一个二维码 —— 在任何装了 Luffa app 的真机上扫一下,就能打开你的互动书。

发给朋友 / 学生 / 同事玩,看他们的反应。

# 心法 · 五条

1. **互动书的核心不是"小屏幕上的书",是"能动手玩的概念"**。如果你只是把书的文字搬上小程序,你浪费了 90% 的潜力。
2. **反预期 ≠ 噱头**。每一条"违反直觉"都要有书里的具体数据 / 案例支撑,否则就是震惊式标题党。读者会一眼看穿。
3. **数学一定先在 Node 里测过再放上线**。AI 生成的数学代码即使语法没错,也可能微妙地错 —— 而错的数学会在用户那里**永远错下去**。
4. **API key 永远不进小程序包**。这一条没有"通融"的余地。所有外部 API 走代理,代理在云端持 key,小程序看不到。
5. **少即是多**。5 个 tabBar、< 500 KB 包体、< 2000 行代码,是 SuperBox 的甜区。功能多不等于体验好,克制是这种轻量平台上最重要的产品智慧。

---

> **最后的话**:这个时代,AI Coder 把"写代码"这件事的成本压得极低 —— 一个原本要团队两周才能做出来的小程序,现在你一个人 12 小时能完成。但**"做什么"的问题 AI 替你不了**。这也是为什么 Phase 0 是整个流程里 AI 含量最低,却是最值钱的一步。
>
> 把这条规律记住,你就掌握了用 AI 做产品的核心:**AI 解决执行,你解决判断**。
